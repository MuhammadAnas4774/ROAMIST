Logic Map — Requirement Traceability Table
Req. ID Class / File Function / Method Implementation Description
F1 LoginActivity.kt btnLogin.setOnClickListener{} Validates email + password fields. On success, creates an explicit Intent(this, MainActivity::class.java) and attaches three extras via putExtra(): EXTRA_USERNAME, EXTRA_SESSION_TOKEN, EXTRA_GREETING. Calls finish() to pop LoginActivity off the back-stack. No SharedPreferences or static variables are used.
Intent Extras No SharedPrefs
F1 DashboardActivity.kt onCreate() — intent.getStringExtra() Host activity reads EXTRA_USERNAME, EXTRA_GREETING, and EXTRA_SESSION_TOKEN from the incoming Intent and displays them in tv_dashboard_greeting and tv_dashboard_username TextViews. Also forwards session extras when launching child activities (Notifications, MainActivity).
Data Receiving Session Forwarding
F1 MainActivity.kt onCreate() — intent.getStringExtra() Receives the same session extras forwarded from LoginActivity. Constructs an Intent to DashboardActivity and re-attaches all three extras, completing the full F1 relay chain: Login → Main → Dashboard.
Intent Chain
F2 Destination.kt writeToParcel() / createFromParcel() Destination is a data class implementing Parcelable. Serialises all 8 fields (id, name, region, description, rating, pricePerNight, category, iconEmoji) via Parcel. The CREATOR companion deserialises them on the receiving end. Zero global state is involved.
Parcelable Model Bundle Safe
F2 DestinationDetailFragment.kt newInstance() / arguments?.getParcelable() Factory method newInstance(destination) is the only way to construct this fragment. It creates a Bundle, calls putParcelable(ARG_DESTINATION, destination), and assigns it to fragment.arguments. In onViewCreated() the Destination is recovered via arguments?.getParcelable(ARG_DESTINATION) and all TextViews are bound in bindDestination().
Bundle Args Factory Pattern
F2 DashboardActivity.kt onDestinationSelected(destination) Implements DestinationListFragment.OnDestinationSelectedListener. When a card is tapped, calls DestinationDetailFragment.newInstance(destination) to package the object into a Bundle, then launches the detail fragment via a Fragment Transaction. This is the bridge between F2 Bundle passing and F4 transaction.
Interface Callback Fragment Launch
F3 DestinationAdapter.kt onCreateViewHolder() / onBindViewHolder() Custom RecyclerView.Adapter with inner DestinationViewHolder class. ViewHolder caches 6 view references (emoji, name, region, category, rating, price) to avoid repeated findViewById calls. Inflates item_destination_card.xml. Maintains two lists: fullList (source of truth) and filteredList (what RecyclerView displays).
Custom Adapter ViewHolder Pattern
F3 DestinationListFragment.kt onViewCreated() — recyclerView setup Sets recyclerView.layoutManager = LinearLayoutManager(requireContext()) and assigns the DestinationAdapter. Seed data is 12 real Pakistani destinations (Hunza, Lahore, Fairy Meadows, Karachi, Skardu, Mohenjo-daro, Nathia Gali, Deosai, Swat, Rohtas Fort, Gwadar, Kalash Valley).
RecyclerView Host 12 Destinations
F4 DashboardActivity.kt showTab() — supportFragmentManager.beginTransaction() Four bottom-nav buttons each call showTab(tabIndex). This calls supportFragmentManager.beginTransaction().replace(R.id.fragment_container, fragment).commit(). No Activity restart occurs — only the fragment content swaps. Handles back-stack awareness via backStackEntryCount. Tabs: Explore (DestinationListFragment), Trips (BookingHistoryFragment), Saved (ExploreGridFragment), Profile (ProfileFragment).
Fragment Replace No Restart
F4 DashboardActivity.kt onDestinationSelected() — addToBackStack() When a destination card is clicked, the detail fragment is loaded with setCustomAnimations(slide_in_left, fade_out...) and .addToBackStack("detail"). The back button pops the fragment rather than finishing the activity, fulfilling the "without restarting the activity" constraint.
Back Stack Custom Animation
F5 DestinationAdapter.kt getFilter() — performFiltering() / publishResults() Implements Filterable. The inner Filter object runs performFiltering() on a background thread — matches query against name, region, and category fields (case-insensitive). publishResults() runs on the main thread, swaps filteredList, and calls notifyDataSetChanged().
Filterable Background Thread
F5 DestinationListFragment.kt etSearch.addTextChangedListener{} Wires the search EditText (et_search_destinations) to the adapter's filter. Every keystroke fires adapter.filter.filter(text), providing real-time live search without any debounce delay. Clearing the field restores the full list instantly.
Live Search Keystroke Listener

Global Constraints — Compliance Evidence

🔒 Constraint 1 — Data Passing via Intent Extras & Bundles Only

Rule: No SharedPreferences, global static variables, or Singletons may be used for inter-screen communication.

Evidence: All session data flows exclusively through Intent.putExtra() / intent.getStringExtra(). The chain is: LoginActivity → putExtra(EXTRA_USERNAME, EXTRA_SESSION_TOKEN, EXTRA_GREETING) → MainActivity → DashboardActivity → NotificationsActivity etc. The SessionIntent.kt helper (forwardSession()) propagates session data between activities without any static field. The Destination model uses Parcelable + Bundle for inter-fragment transfer. No single SharedPreferences.edit() call exists in the entire codebase.

🧩 Constraint 2 — Modular UI via Fragments (Activities as Containers Only)

Rule: Primary UI content must live in Fragments. Activities are navigation coordinators.

Evidence: DashboardActivity contains a single fragment_container FrameLayout. All tab content is rendered by 4 Fragment classes: DestinationListFragment, BookingHistoryFragment, ExploreGridFragment, ProfileFragment. Destination detail is DestinationDetailFragment. The activity itself contains no list data or UI logic — it only calls supportFragmentManager.beginTransaction().

📋 Constraint 3 — Dynamic Lists via RecyclerView + Custom Adapter + ViewHolder

Rule: All lists must use RecyclerView with a custom Adapter and ViewHolder.

Evidence: DestinationAdapter extends RecyclerView.Adapter<DestinationAdapter.DestinationViewHolder>. The inner DestinationViewHolder caches view references for emoji, name, region, category, rating, and price TextViews. onCreateViewHolder() inflates item_destination_card.xml. onBindViewHolder() calls holder.bind(filteredList[position]).

📁 Constraint 4 — Clean Architecture (Proper Folder / Package Organization)

Rule: Code must be organized into proper packages: Activities, Fragments, Adapters, Models.

Evidence: The project (com.example.roamist) is split into four sub-packages: .activities (16 classes), .fragments (5 classes), .adapters (1 class), .models (1 class). XML layouts are in res/layout/ with naming conventions (activity_*, fragment_*, item_*). Separation of concerns is maintained — no business logic in XML, no UI binding in models.
